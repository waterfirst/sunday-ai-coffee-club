## 세미나 A: 코딩 에이전트 토큰 효율 — 설정만 바꿔도 비용이 반으로

### 배경

- Claude Opus 4.7 출시 (4/17) → 토큰 소모량 1.35~1.5배 증가
- Pro 플랜도 토큰 리밋에 자주 걸림
- 설정만 바꾸면 성능 저하 없이 토큰 낭비를 크게 줄일 수 있음

### 발표 내용 (30~40분)

#### 1. 토큰이 새는 3가지 경로

**경로 1: 매 턴 자동 주입 텍스트**
- `includeGitInstructions: true` (기본) → 매 턴 git 지시문 수백 토큰
- IDE 자동 연결 (`autoConnectIde: true`) → 불필요한 컨텍스트
- MCP 서버 도구 정의 → Canva, Asana, Linear 등 연결된 커넥터마다 수천 토큰
- attribution 헤더 → 소량이지만 누적

**경로 2: 긴 툴 호출 출력**
- bash 명령어 결과가 무제한으로 히스토리에 쌓임
- 대형 파일 전체 읽기 → 수만 토큰 낭비
- MCP 도구 응답이 과도하게 큰 경우

**경로 3: 외부 연결 추가 호출**
- 웹 검색 자동 호출
- IDE 컨텍스트 동기화
- claude.ai MCP 커넥터 (Gamma, Indeed, Intercom 등)

#### 2. Claude Code 레버 6개

```json
// ~/.claude/settings.json 추천 설정
{
  "preferences": {
    "includeGitInstructions": false,
    "autoConnectIde": false
  },
  "env": {
    "BASH_MAX_OUTPUT_LENGTH": "30000",
    "CLAUDE_CODE_FILE_READ_MAX_OUTPUT_TOKENS": "20000",
    "MAX_MCP_OUTPUT_TOKENS": "50000"
  }
}
```

각 설정별 절약 효과와 트레이드오프 설명

#### 3. 캐싱 효율 연구

- claude-code-cache-fix 레포 소개
- 캐시 히트율이 비용에 미치는 영향
- Opus 4.7 모델 전환 시 캐시 무효화 → 초기 비용 스파이크

#### 4. 실습: 내 설정 점검 프롬프트

참석자 각자의 Claude Code 설정을 토큰 효율 분석 프롬프트로 점검

## 세미나 B: AI 메모리 시스템의 진화 — LLM Wiki v2와 DOT Studio

### 배경

- Andrej Karpathy의 LLM Wiki → v2로 진화
- RAG만으로는 부족, LLM이 직접 지식을 관리하는 시대
- DOT Studio: AI 에이전트 오케스트레이션의 Figma

### 발표 내용 (30~40분)

#### 1. LLM Wiki v2 — 알아서 강해지고 잊어버리는 위키

**원본의 한계:**
- 위키에 한 번 적힌 내용은 시간이 지나도 동일한 신뢰도
- 새 정보가 들어와도 옛 내용은 방치

**v2가 메운 것:**

**(a) 신뢰도 점수 (Confidence Scoring)**
- 모든 사실에 메타데이터 부착
- 출처 수, 최신성, 모순 여부에 따라 점수 오르내림
- 에빙하우스 망각 곡선: 오래 안 쓰면 흐려지고, 재확인되면 또렷해짐

**(b) 기억의 4단계**

| 단계 | 이름 | 설명 | 수명 |
|------|------|------|------|
| 1 | Working | 방금 관찰한 것 | 매우 짧음 |
| 2 | Episodic | 한 세션 요약 | 세션 단위 |
| 3 | Semantic | 여러 세션에서 검증된 사실 | 장기 |
| 4 | Procedural | 반복 작업 패턴 | 가장 오래 생존 |

**(c) 지식 그래프**
- 마크다운 → 노드 + 타입 + 관계가 붙은 그래프
- 검색: BM25 + 벡터 유사도 + 그래프 탐색 동시

**(d) 자가 치유**
- 고아 페이지 수정, 깨진 참조 복구
- 모순 발견 시 출처 권위 + 시간 순서로 해결 제안

#### 2. 실제 적용 사례: Claude Code 메모리에 v2 개념 이식

우리 모임에서 실제로 사용 중인 Claude Code 메모리 시스템에 적용한 사례:

```markdown
---
name: 렛유인 KDC 제약조건
type: project
confidence: 0.95        ← 신뢰도 점수
sources: 2              ← 출처 수
lastVerified: 2026-04-19 ← 마지막 확인일
tier: semantic           ← 기억 계층
supersedes: null         ← 대체 관계
---
```

- 단순 마크다운 → 구조화된 메타데이터 부착
- 미래 세션에서 어떤 정보가 더 신뢰할 만한지 즉시 판단 가능

#### 3. DOT Studio — AI 에이전트의 Figma

**핵심 구조:**
```
Tal(정체성) + Dance(스킬) + 모델 + 도구 = Performer(에이전트)
여러 Performer + choreography = Act(워크플로우)
```

**특징:**
- Figma처럼 캔버스에서 드래그/드롭으로 에이전트 배치
- 에이전트 간 관계, 협업 규칙, wake-up 신호 정의
- OpenCode 위에서 실행
- npm install -g dot-studio 로 바로 사용

**왜 중요한가:**
- 멀티에이전트 시대의 표준 도구가 될 가능성
- "코드로 에이전트를 정의하는 시대" → "시각적으로 안무하는 시대"
- 렛유인 강의의 고급 주제로도 활용 가능

#### 4. 데모 (가능하면)

- DOT Studio 설치 및 간단한 2-Performer Act 생성
- Claude Code 메모리 시스템의 confidence/tier 동작 시연
